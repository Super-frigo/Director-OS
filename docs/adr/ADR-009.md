# ADR-009: Agent Implementation Strategy

Date: 2026-07-13

## Status

Accepted

## Context

Director OS currently operates as a deterministic batch compiler: the user writes a
Project YAML by hand, runs `load → plan → compile` via CLI, and receives a platform
prompt.  This works as a proof-of-concept but falls short of the system defined in
[DOCS_SPEC.md](../DIRECTOR_SPEC.md) and [WORKFLOW_SPEC.md](../WORKFLOW_SPEC.md).

The gap between specification and implementation:

```text
Spec:       User → Director → Agents → Project → Compiler → Prompt
               ↻ 8-state Creative Cycle, multi-Cycle continuity

Current:     CLI → Director → Engine → Compiler → Prompt
               No state machine, no dialogue, no multi-Cycle editing
```

Four capabilities are missing:

1. **Project 写回（Persistence）** — `load_project()` exists but there is no
   `save_project()`.  Agents have no way to write their results.
2. **状态机（State Machine）** — The 8 states defined in `DIRECTOR_STATE_MACHINE.md`
   (IDLE → UNDERSTAND → CLARIFY → PLAN → DESIGN → VALIDATE → COMMIT → COMPILE)
   are documented but not implemented.  `Director` is stateless.
3. **Agent 协议（Agent Protocol）** — WORKFLOW_SPEC defines Story Agent, Camera Agent,
   Continuity Agent, etc., but no interface, input/output schema, or dispatch mechanism
   exists.
4. **对话层（Dialogue Layer）** — SYSTEM_PROMPT.md defines a Director persona,
   but there is no conversation context, intent parsing, or multi-turn memory.

Attempting to implement all four at once would be risky and difficult to debug.

## Decision

**Phase 5 Agent 化采用渐进式三步走：**

```text
Step 1 ──► Step 2 ──► Step 3
(Current)   (Next)     (Future)

Step 1                          Step 2                          Step 3
Project 读写闭环               状态机 + MCP Server             多 Agent 拆分
save_project / dump_project    8-state Creative Cycle          Story / Camera /
auto-version / History          Director as MCP tool           Continuity Agents
                                Validated before commit        Director as orchestrator
```

**Each Step is independently usable.** Step 2 does not require Step 3.
Step 1 does not require Step 2. Each delivers value on its own while building
toward the complete vision.

### Step 1: Project 读写闭环（数据层）

**Target output:** `Director.save_project(path, message)` produces a YAML file that
`load_project()` can read back with full fidelity. History entries are automatically
appended.

**Components:**
- `dump_project(Project) → YAML string` — serializer, symmetric to the existing loader
- `save_project_file(Project, path)` — writes Markdown-enveloped YAML to disk
- `Director.save_project(path, message)` — validates before save, auto-versions,
  appends `HistoryEntry`, then writes
- CLI `save` subcommand for direct invocation

**Design constraints (from existing ADRs):**
- ADR-001 → Project IS the source of truth. No side-channel writes.
- ADR-002 → Compiler remains isolated. Save never generates prompts.
- Principle 6 → Validation blocks commit. `save_project()` calls `validate()` first.
- Principle 7 → Immutable history. Every save appends a versioned `HistoryEntry`.

### Step 2: 状态机 + MCP Server（控制层）

```text
User utterance ──► Director Agent (LLM with MCP tools)
                       │
                       ├── director.new_project(title, premise)
                       ├── director.understand(utterance)
                       ├── director.clarify(question)
                       ├── director.plan()
                       ├── director.design()
                       ├── director.validate_project()
                       └── director.save_project(path, message)
```

The state machine is implemented as a guard layer: each state transition is validated
(you cannot jump from CLARIFY to COMPILE), and the Director's system prompt enforces
the 7-stage pipeline from SYSTEM_PROMPT.md.

The MCP Server exposes `Director` methods as MCP tools, following the same protocol
used by Knowledge Resolution (ADR-008). Any MCP-capable client (Claude Desktop/ZCode)
can connect as the Director.

**Components:**
- `DirectorState` enum and transition validation
- `Director.understand()` / `.clarify()` / `.plan()` / `.design()` wrappers
- `mcp_server.py` — MCP tool registration
- Updated SYSTEM_PROMPT.md → MCP tool descriptions

### Step 3: 多 Agent 拆分（协作层）

```text
Director (编排者,唯一对用户)
  ├── Story Agent     → story_beats / emotional arc / structure
  ├── Character Agent → character sheets / continuity lock
  ├── Camera Agent    → shot list / framing / lens / movement
  └── Continuity Agent →跨 Cycle 一致性校验
```

Each Agent:
- Has its own system prompt + knowledge domain binding (e.g., Camera Agent binds to
  camera/lighting/composition knowledge providers)
- Reads a narrow slice of the Project (its "module")
- Outputs structured modification proposals: `{module, field, action, data}`
- Never talks directly to the user (DIRECTOR_SPEC line 14-15: "用户永远不会直接与
  Story Agent 交互")

Director synthesizes all Agent outputs, runs Continuity checks, and presents a
unified result. All Agents communicate through the Project (SSOT), never directly
with each other. This follows Principle 1 and prevents agent-agent inconsistency.

**Components:**
- `AgentBase` protocol defining `agent_name`, `bound_knowledge_domains`,
  `read_project_slice(project)`, `propose_changes(project, intent)`
- One implementation per Agent type
- `Director.run_agent_cycle()` — dispatches to Agents and merges results

## Alternatives Considered

### Alternative 1: 一次性实现所有 Agent

**Rejected.** 8 parallel Agents + state machine + dialogue layer would create too many
concurrent unknowns. Debugging agent-to-agent interaction without a working state
machine or persistence layer would be impossible to reason about.

### Alternative 2: 先做 UI 再做 Agent

**Rejected.** A web UI without Agent intelligence is just a YAML editor. The creative
reasoning pipeline (understand → plan → design) is the core value proposition.
Building a UI before the Agent layer exists creates a dependency inversion that makes
iterating on the core logic harder.

### Alternative 3: 只在 System Prompt 里实现（不加 MCP）

**Rejected.** The Director's 7-stage pipeline generates large amounts of structured
data (story beats, shot lists, character sheets). Keeping this all in the LLM's
context window would quickly exhaust token limits and make multi-Cycle continuity
impossible. MCP tools give the LLM a "long-term memory" through the Project file.

## Consequences

### Positive

- Step 1 delivers a working save/load loop immediately, enabling the Project to be
  created and modified through code rather than only by hand-editing YAML
- Each Step validates the assumptions of the next: the serializer must work before
  the state machine can commit; the state machine must work before we can add
  multiple agents
- MCP Server in Step 2 makes the system instantly usable from any MCP-capable LLM
  client (Claude Desktop, ZCode, Cursor, etc.)

### Negative

- The Project YAML format is verbose for hand-editing (but this is intentional —
  the Agent is supposed to be the primary writer)
- Lighting data lost on round-trip requires extending the LightningSetup model
  (see Implementation Notes)
- Markdown envelope reconstruction may not perfectly match the original (section
  headers are regenerated from canonical names, not preserved from input)

### Neutral

- The `camera_raw` pattern (already present) must be extended to ensure full
  round-trip fidelity for nested sub-dicts
- History entries are string-based, not structural diffs — rollback is currently
  manual (revert to a prior version in git)

## Implementation Notes

### Step 1 file layout

```
docs/adr/ADR-009.md              ← this file
director_os/models/project.py    ← LightingSetup +2 fields (position, intensity)
director_os/loader.py            ← + dump_project() / save_project_file()
director_os/director.py          ← + save_project()
director_os/cli.py               ← + save subcommand
tests/test_save.py               ← round-trip tests
```

### Round-trip fidelity guarantees

- `camera_raw` on Shot preserves the full camera sub-dict from input YAML
- `LightingSetup` now stores position + intensity (previously dropped)
- Shared fields (subject, color, audio, emotion) are reconstructed as sub-dicts
- Default value suppression: empty strings, zeros, false, empty lists/dicts are
  omitted from output YAML to keep files readable
- YAML key canonicalization: `shot_id` (not `id`), `beat_ref` (not `beat`),
  `visual_language` (not `visual`), `output` (not `output_profile`)

### Known data loss (accepted)

- `story.protagonist`, `story.climax`, `story.ending` etc. — narrative notes in
  `the_hanging.md` that exist outside the typed Story schema. These are dropped
  by the loader and not recoverable. Future Story schema extension may add them.
- `visual.camera` at the VisualLanguage level — camera defaults nested under visual
  in the_hanging.md. Not stored by the VisualLanguage dataclass. Each shot already
  carries its own camera settings.

## Summary

Phase 5 delivers value incrementally:
- Step 1 makes Project a live document that Agents can write back
- Step 2 makes Director an interactive MCP tool with state-machine-gated flows
- Step 3 distributes creative reasoning across specialist Agents while Director
  remains the single user-facing interface

Each Step is designed to be independently useful and to validate assumptions for
the next. The architecture principles from Phase 1-2 (SSOT, Compiler isolation,
strict layering) remain the foundation — Agent 化 is about exposing those
principles through a conversational interface, not replacing them.
