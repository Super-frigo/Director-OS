# ADR-002: Compiler Must Be Isolated

Date: 2026-07-11

## Status

Accepted

## Context

Director OS 的目标不是生成 Prompt，而是建立一个面向 AI 视频创作的结构化电影操作系统。

在系统中：

```
Creative Intent
        ↓
Director
        ↓
Project
        ↓
Compiler
        ↓
AI Video Model
```

Project 表示电影本身，包括：

* Story
* Characters
* Scenes
* Shots
* Camera Language
* Lighting
* Composition
* Visual Style

这些概念属于电影创作领域，而不是某一个 AI 视频模型的输入格式。

然而，不同 AI 视频平台拥有不同的 Prompt 结构：

* 不同字段命名
* 不同能力边界
* 不同控制参数
* 不同模型特性
* 不同版本变化

如果平台 Prompt 结构直接进入 Project 层，系统会逐渐退化为：

```
Project = Prompt Template
```

最终导致：

* 创作逻辑绑定具体模型
* 平台变化导致项目结构变化
* 导演无法保持长期创作资产
* 多模型协作无法实现

因此 Compiler 必须成为独立边界。

## Decision

Compiler 必须被设计为 Director OS 的最后一层，并保持完全隔离。

Compiler 的唯一职责：

> 将中立的 Project Representation 编译为目标 AI 平台可以理解的生成指令。

Compiler 不拥有：

* 创作决策权
* 故事修改权
* 角色设计权
* 镜头设计权
* 风格定义权

Compiler 只能进行：

```
Project Model
      ↓
Compiler
      ↓
Platform Specific Output
```

转换。

## Architectural Rules

### Rule 1: Project Never Knows Compiler

Project schema 不允许出现：

* seedance_prompt
* veo_prompt
* kling_parameters
* runway_settings

等平台相关字段。

Project 描述：

"电影是什么"

而不是：

"模型需要什么 Prompt"

---

### Rule 2: Compiler Is Replaceable

任何 Compiler 都应该可以被替换：

```
Project
   |
   +-- Seedance Compiler
   |
   +-- Veo Compiler
   |
   +-- Kling Compiler
   |
   +-- Runway Compiler
```

增加新的 AI 平台，不应该修改：

* Project Schema
* Director Logic
* Library System
* Review System

---

### Rule 3: Compiler Has No Memory

Compiler 不保存：

* 创作状态
* 导演意图
* 项目历史

所有信息必须来自 Project。

Compiler 是纯转换层：

```
Output = Compiler(Project)
```

同一个 Project 在同一个 Compiler 规则下，应产生可预测结果。

---

### Rule 4: Platform Optimization Happens Inside Compiler

平台特性优化必须发生在 Compiler 内部。

例如：

同一个镜头：

```
A lonely astronaut walking through a desert.
```

Project 层描述：

```
Scene:
- Subject: astronaut
- Environment: desert
- Emotion: isolation
- Camera: wide cinematic shot
```

Compiler 决定：

Seedance:

```
duration=10s
camera_motion=slow_push
```

Veo:

```
cinematic movement
high temporal consistency
```

这些差异不能反向污染 Project。

## Alternatives Considered

### Alternative 1: Store Platform Prompts Directly In Project

Rejected.

原因：

* 项目被模型格式绑定
* 无法跨平台
* 长期资产价值降低

---

### Alternative 2: Let Director Generate Platform Prompts Directly

Rejected.

原因：

Director 的职责是导演，而不是 Prompt Engineer。

如果 Director 直接生成 Prompt：

```
Director = Prompt Generator
```

系统会失去创作抽象层。

---

### Alternative 3: Create Separate Prompt Templates Only

Rejected.

原因：

Template 只能解决格式问题，无法解决语义转换问题。

Compiler 需要理解：

* 镜头语言
* 视觉风格
* 平台能力

而不仅是字符串替换。

## Consequences

### Positive

* Project 成为长期稳定资产
* 支持多个 AI 视频模型
* 新模型接入成本降低
* 创作逻辑不会被供应商控制
* Director 保持导演角色

### Negative

* Compiler 开发复杂度增加
* 需要维护多个平台适配层
* 需要持续跟踪模型能力变化

### Neutral

Compiler 会成为系统的重要工程模块，但不会成为创作核心。

## Implementation Notes

未来实现应保持：

```
director-os/

├── projects/
│
├── engines/
│
├── libraries/
│
└── compilers/
    ├── seedance/
    ├── veo/
    ├── kling/
    └── runway/
```

Compiler 层应依赖：

* Project Schema
* Library Resolution Result
* Engine Output

但不能反向依赖：

* Director State
* User Conversation
* Platform API Details

## Summary

Compiler 是 Director OS 的出口，而不是核心。

Director OS 负责创造电影。

Compiler 负责让不同 AI 模型理解这部电影。

两者必须保持隔离。
