# ADR-008: Knowledge Resolution Architecture

Date: 2026-07-12

## Status

Accepted

## Context

ADR-004 建立了 Library Driven Knowledge System，将电影知识存储为结构化 YAML 条目。

但 Library-as-storage 模型存在根本局限：

> **电影知识的规模和演化速度远超静态文件系统能承载的范围。**

具体问题：

1. **知识覆盖永远不完整。** 无论编写多少 YAML 条目，总会有 Library 未覆盖的电影技法、叙事结构、视觉风格。

2. **知识维护成本线性增长。** 每新增一个知识条目都需要人工编写、审核、维护。

3. **LLM 自身是巨大的电影知识库。** GPT-5、Claude、Gemini 等模型已在训练中吸收了海量电影理论、摄影技法、导演风格知识。把这些知识重新手写为 YAML 是在重复已存在的工作。

4. **Library 的"结构化"优势不应被抛弃。** 确定性、可审计、可版本控制——这些是 Library 的核心价值，不能用纯 LLM 调用替代。

核心矛盾：

```text
Library 的精确性  vs  LLM 的覆盖广度
```

Director OS 需要一种架构，同时利用两者的优势。

## Decision

**用 Knowledge Resolution Architecture 取代 Library-as-storage。**

Director OS 不再定义"知识存储在哪里"，而是定义：

> **知识如何被请求、解析和裁决。**

```text
旧模型（Library-as-storage）：
  Engine → Library (YAML files) → Knowledge

新模型（Knowledge Resolution）：
  Engine → Knowledge Request → Resolution Pipeline → Resolved Knowledge
                                       │
                    ┌──────────────────┼──────────────────┐
                    │                  │                  │
               Local Rules        Cached LLM        Future Providers
              (deterministic)   (deterministic     (databases, APIs,
                                via caching)        domain models)
```

## Provider 类型

### Provider 1: Local Rules（本地规则）

本地规则是**确定性知识源**，来自现有 Library 条目的迁移。

```yaml
provider: local_rules
characteristics:
  - 100% 确定性：相同查询 → 相同结果
  - 零外部依赖
  - 可 Git 版本控制
  - 可人工审计和修正
```

本地规则是系统的**确定性基座**。当其他 Provider 不可用或结果不可信时，本地规则是始终可用的回退。

### Provider 2: Cached LLM（缓存 LLM）

LLM 拥有远超本地规则的知识广度，但不能直接引入非确定性。

```yaml
provider: cached_llm
mechanism:
  - Engine 发起知识请求
  - Resolution Pipeline 检查缓存
  - 缓存命中 → 直接返回（确定性）
  - 缓存未命中 → 调用 LLM → 写入缓存 → 返回
characteristics:
  - 缓存命中时 100% 确定性
  - 覆盖本地规则无法覆盖的知识
  - 缓存文件可纳入版本控制
  - 缓存预热：可在 CI 中预先填充常用查询
```

关键设计决策：**LLM 在 Engine 阶段调用，不在 Compiler 阶段调用。** Compiler 只看到已解析的确定性知识。

### Provider 3: Future Providers（未来 Provider）

架构预留扩展点：

```yaml
future_providers:
  - domain_databases:     # 专业电影数据库
  - specialized_models:   # 电影领域微调模型
  - community_knowledge:  # 社区贡献的知识条目
  - api_providers:        # 第三方知识 API
```

任何实现了 Resolution Provider 接口的组件都可以接入。

## 核心规则

### Rule 1: Compiler Never Calls LLM（Compiler 永不调用 LLM）

```
正确：
  Engine → Resolution (calls LLM if needed) → Resolved Knowledge → Compiler

错误：
  Compiler → LLM → Prompt
```

Compiler 接收的是**已解析的确定性知识**。从 Compiler 的视角，知识来源是透明的——它不知道也不需要知道某条知识来自本地规则还是缓存 LLM。

这与 ADR-002 一脉相承：Compiler 是纯转换层。

### Rule 2: Engine Requests, Resolution Resolves（Engine 请求，Resolution 解析）

```text
Engine:    "我需要关于 X 的知识"
Resolution: "这是关于 X 的知识（来源：Y，可信度：Z）"
Engine:    "基于此知识做出创意决策"
```

Engine 不拥有知识，不存储知识，只**请求和消费**知识。Resolution 层负责"从哪里获取"和"如何裁决"。

### Rule 3: Determinism via Caching（通过缓存保证确定性）

LLM 知识引入的唯一非确定点是**首次调用**。之后所有相同查询都命中缓存。

缓存策略：

```yaml
cache:
  key: "hash(query + context_signature)"
  invalidation: "manual + version-tagged"
  storage: "version-controlled cache files"
  warmup: "CI pipeline pre-fills common queries"
```

缓存文件纳入 Git 版本控制。这意味着：
- 团队共享相同的解析结果
- 可以回滚到任何版本的缓存
- CI 可以验证缓存一致性

### Rule 4: Local Rules Are the Fallback（本地规则是回退基座）

```text
Resolution 优先级：
  1. Cached LLM（如果缓存命中）
  2. Local Rules（始终可用）
  3. Live LLM（仅在允许且缓存未命中时）

底线：
  如果 LLM 不可用，系统降级到纯本地规则，仍然可工作。
  如果 LLM 可用，系统获得更广的知识覆盖。
```

### Rule 5: Source Traceability（来源可追溯）

每条解析后的知识必须标注来源：

```yaml
resolved_entry:
  content: "Wide shots with negative space express isolation"
  sources:
    - provider: local_rules
      path: "libraries/composition/entries/comp_negative_space.yaml"
    - provider: cached_llm
      cache_key: "abc123def456"
      model: "gpt-5"
      timestamp: "2026-07-12T10:00:00Z"
  confidence: 0.92
```

可追溯性确保了：
- 可以审计任何创意决策的来源
- 可以识别和修正低质量知识
- 可以对比不同 Provider 的质量

## 与现有架构的关系

### 对 ADR-004 的影响

ADR-004（Library Driven Knowledge System）被本 ADR **取代**。

Library 条目迁移为 Local Rules Provider 的输入数据。Library 的概念不再作为独立架构层存在，而是成为 Knowledge Resolution 的一个 Provider。

### 对 ADR-003 的影响

ADR-003（Engine Layer）中 Engine 的职责从"查询 Library"更新为"通过 Resolution 请求知识"。Engine 仍然是知识的消费者，但现在知识来源是透明的。

### 对 ADR-002 的影响

ADR-002（Compiler Must Be Isolated）的核心原则不变。Compiler 的确定性由 Resolution 层的缓存机制保证——LLM 调用发生在 Compiler 之前。

## Alternatives Considered

### Alternative 1: Keep Library Only（保持纯 Library）

**Rejected.** 知识覆盖永远不完整，维护成本线性增长。

### Alternative 2: Let Compiler Call LLM（让 Compiler 调用 LLM）

**Rejected.** 违反 ADR-002。Compiler 必须保持确定性。如果 Compiler 调用 LLM，相同 Project 在不同时间可能产生不同 Prompt，破坏整个系统的可预测性。

### Alternative 3: LLM-Only, No Local Rules（纯 LLM，无本地规则）

**Rejected.** 完全放弃确定性不可接受。电影制作需要可复现的结果。本地规则是确定性基座。

### Alternative 4: Hybrid Without Caching（混合但无缓存）

**Rejected.** LLM 的非确定性会污染整个下游管线。缓存是将 LLM 知识"确定性化"的关键机制。

## Consequences

### Positive

- 知识覆盖不再受限于手写条目数量
- 新知识可以零成本获取（LLM 查询 vs 手写 YAML）
- 本地规则保留确定性和可审计性
- 架构天然支持未来知识源扩展
- Compiler 保持确定性——所有 LLM 调用被封在 Resolution 层
- 知识来源可追溯，质量可审计

### Negative

- 系统复杂度增加（新增 Resolution Pipeline）
- 缓存管理需要工程规范
- LLM 知识质量需要审核机制
- 本地规则和 LLM 知识的合并策略需要设计

### Neutral

- Library 条目不会消失——它们迁移为 Local Rules Provider 的数据
- 现有 Library 实现（composition query 等）作为 Local Rules 的参考实现继续存在
- 这是一个渐进式迁移，不是一次性重写

## Implementation Notes

### 目录结构变更

```text
旧结构：
  libraries/           ← 知识存储
    composition/
    camera/
    lighting/
    ...

新结构：
  knowledge/           ← Knowledge Resolution 层
    providers/
      local_rules/     ← 原 libraries/ 迁移至此
        composition/
        camera/
        lighting/
        ...
      llm_cache/       ← LLM 响应缓存
    resolver.py        ← Resolution Pipeline
    providers.py       ← Provider 接口定义
```

### Provider 接口

```python
class KnowledgeProvider:
    """知识提供者接口"""

    def resolve(self, request: KnowledgeRequest) -> list[KnowledgeEntry]:
        """解析知识请求，返回匹配条目列表"""
        ...

    @property
    def name(self) -> str: ...

    @property
    def is_deterministic(self) -> bool: ...
```

### 渐进迁移策略

1. Phase A: 定义 Provider 接口和 KnowledgeRequest/KnowledgeEntry 数据模型
2. Phase B: 将现有 Library 条目迁移为 LocalRulesProvider
3. Phase C: 实现 CachedLLMProvider（含缓存层）
4. Phase D: 实现 Merge & Rank 逻辑
5. Phase E: 更新 Engine 使用 Resolution API 替代直接 Library 查询

每个 Phase 都可以独立交付，系统在每一步都保持可用。

## Summary

Director OS 不再问"知识存储在哪里"，而是问：

```text
"当 Engine 需要电影知识时，系统如何以确定性的方式提供最佳可用的知识？"
```

答案：

```text
Knowledge Resolution Architecture

  Local Rules（确定性基座）
       +
  Cached LLM（广度覆盖）
       +
  Future Providers（架构可扩展）
       =
  确定性的、可扩展的、来源可追溯的电影知识系统
```

Library 是知识的**一种存储形式**。

Knowledge Resolution 是知识的**获取和裁决机制**。

前者是 What，后者是 How。

Director OS 的职责从管理知识存储，演进为管理知识解析。
