# ADR-006: Canonical Project Representation

Date: 2026-07-11

## Status

Accepted

## Context

Director OS 的核心目标是成为 AI-native filmmaking operating system。

此前架构已经建立：

* ADR-001:
  Project is the Single Source of Truth

* ADR-003:
  Engine Layer provides creative reasoning

* ADR-004:
  Library System provides reusable knowledge

* ADR-005:
  Review System creates feedback loops

这些系统都依赖一个核心对象：

```text
Project
```

Project 是 Director OS 中最重要的数据结构。

它必须同时满足：

* 人类导演可以理解
* AI 系统可以解析
* 工具链可以处理
* 长期保存稳定
* 支持版本管理

如果没有统一表示：

不同模块会产生自己的数据格式：

```text
Story Format

Scene Format

Shot Format

Engine Format

Compiler Format
```

最终导致：

```text
No Single Source Of Truth
```

系统无法保持一致。

因此 Director OS 需要定义：

Canonical Project Representation。

## Decision

Director OS 采用 Canonical Project Representation 作为所有创作数据的标准表示。

定义：

> Canonical Project Representation is the stable, structured representation of a film project that all Director OS components understand.

架构：

```text
Human Intent

      ↓

Canonical Project Representation

      ↓

Engine

      ↓

Compiler

      ↓

AI Models
```

所有核心模块必须围绕 Project Representation 工作。

## Design Principles

## 1. Project Representation Must Be Human Readable

导演和创作者需要直接理解项目内容。

例如：

```yaml
project:
  title: "The Last Signal"

  characters:
    - name: "Maya"

  scenes:
    - id: scene_001
      emotion: isolation
```

创作者不应该依赖数据库界面或内部 API 才能理解作品。

---

## 2. Project Representation Must Be Machine Interpretable

系统组件需要能够：

* parse
* validate
* transform
* analyze

例如：

```text
Project

 ↓

Schema Validation

 ↓

Engine Processing
```

---

## 3. Project Representation Must Be Platform Independent

Project 不允许包含：

* Model-specific prompts
* API parameters
* Vendor-specific fields

错误：

```yaml
generation:
  veo:
    camera_strength: 0.8
```

正确：

```yaml
camera:
  movement: slow_push
```

平台转换由 Compiler 完成。

---

## 4. Project Representation Must Be Versioned

电影项目会持续变化。

因此：

```yaml
project:
  version: "1.0"
```

必须支持：

* schema evolution
* migration
* backward compatibility

长期目标：

一个十年前创建的项目，仍然可以被未来 Director OS 打开。

## YAML Decision

当前阶段：

Director OS 使用 YAML 作为 Canonical Project Representation 的主要序列化格式。

原因：

### Human Friendly

相比 JSON：

* 更容易阅读
* 更适合编辑
* 更接近创作者思维

### Structured

相比纯文本：

* 支持层级结构
* 支持 schema validation
* 支持自动处理

### Version Control Friendly

适合：

* Git
* Diff
* Review
* Collaboration

因此：

```text
Canonical Representation
          +
          |
          ↓
        YAML
```

但：

YAML 不是架构本身。

它只是当前实现。

## Architectural Rules

## Rule 1: All Components Consume Project Representation

Engine:

```text
Project
 ↓
Reasoning
```

Compiler:

```text
Project + Engine Output
 ↓
Translation
```

Review:

```text
Project + Result
 ↓
Evaluation
```

所有模块围绕统一表示。

---

## Rule 2: No Hidden Project State

系统禁止：

```text
Project

+
Hidden Database State

+
Temporary AI Memory
```

组成真实项目。

项目必须可以独立保存和恢复。

---

## Rule 3: Schema Evolution Must Be Managed

任何结构变化必须：

* versioned
* documented
* migrated

例如：

```text
Project Schema v1

        ↓

Migration

        ↓

Project Schema v2
```

---

## Rule 4: Representation Is Independent From Storage

未来 Project 可以存储于：

```text
YAML File

Database

Graph Database

Cloud Storage

Knowledge System
```

但内部语义保持一致。

## Alternatives Considered

## Alternative 1: Use Database As Primary Project Format

Rejected.

原因：

* 不适合创作者直接编辑
* 降低可移植性
* 增加系统依赖

---

## Alternative 2: Use Plain Text Documents

Rejected.

原因：

* 缺少结构
* 难以自动处理
* 无法支持复杂电影项目

---

## Alternative 3: Store Model Prompts Directly

Rejected.

原因：

违反：

* ADR-001
* ADR-002

会导致项目绑定 AI 平台。

## Consequences

## Positive

* 建立稳定系统核心
* 支持长期项目资产
* 支持多工具协作
* 支持未来技术变化

## Negative

* 需要维护 Schema
* 需要处理版本迁移
* 初期设计成本增加

## Neutral

YAML 会成为当前阶段主要开发格式，但未来可以替换。

## Implementation Notes

推荐结构：

```text
director-os/

├── projects/

│   └── example_project.yaml

├── schemas/

│   └── project.schema.yaml

├── migrations/

│   ├── v1_to_v2/

│
├── engines/

├── libraries/

├── compilers/

└── reviews/
```

Example:

```yaml
project:
  schema_version: 1.0

  metadata:
    title: Example Film

  story:
    premise: ""

  characters: []

  scenes: []

  visual_language: {}

  production_intent: {}
```

## Summary

Director OS requires a stable representation of creative intent.

The Project Representation is the foundation connecting:

* Human creativity
* Engine reasoning
* Knowledge libraries
* Compiler translation
* Review feedback

The complete Genesis Architecture becomes:

```text
                 Human Director

                       ↓

          Canonical Project Representation

                       ↓

                    Engine

              ↙                 ↘

        Libraries              Review

                       ↓

                  Compiler

                       ↓

                 AI Models
```

The representation belongs to the creator.

The models are only execution tools.

Director OS preserves creative intent beyond any individual AI platform.
