# ADR-001
# Project is the Single Source of Truth

Status: Accepted

Version: 1.0

Date: 2026-06-11

Owner: Director OS Architecture

---

# Context

Traditional AI prompt workflows treat the prompt as the primary artifact.

A user repeatedly edits prompts until the generated result is satisfactory.

This approach creates several problems:

- Creative decisions are embedded inside prompts.
- Story, camera, lighting and character information become duplicated.
- Prompt revisions overwrite previous creative intent.
- Supporting multiple video models requires rewriting prompts.
- Long-term project maintenance becomes impossible.

Director OS is intended to solve these problems.

Therefore the system requires a persistent canonical representation of the film.

---

# Decision

Director OS adopts **Project** as the Single Source of Truth (SSOT).

Every creative decision must be stored inside Project.

No subsystem may maintain its own independent version of project data.

Prompts are generated from Project.

Prompts are never treated as source data.

---

# Consequences

## Positive

Project becomes model-independent.

Creative information exists only once.

Multiple compilers can share identical source data.

Project history becomes versionable.

Creative collaboration becomes possible.

Future tools can operate on Project directly.

---

## Negative

Compiler complexity increases.

Project schema becomes more important.

Project validation becomes mandatory.

Migration between schema versions must be supported.

---

# Scope

This decision applies to:

- Story
- Characters
- Camera
- Lighting
- Production Design
- Audio
- Story Beats
- Shot List
- Continuity
- Constraints

All creative assets belong to Project.

---

# Explicit Non-Goals

Project does NOT contain:

- Seedance Prompt
- Veo Prompt
- Kling Prompt
- Model Parameters
- Temporary Prompt Engineering Tricks
- Compiler-specific Syntax

These belong to Compiler.

---

# Architectural Rule

Every subsystem follows:

User Idea

↓

Creative Brief

↓

Project

↓

Story Bible

↓

Shot Bible

↓

Compiler

↓

Prompt

No subsystem may bypass Project.

---

# Responsibilities

Director

Creates and updates Project.

Story Engine

Reads Project.

Updates Story.

Camera Engine

Reads Project.

Updates Camera Language.

Continuity Engine

Reads Project.

Maintains consistency.

Compiler

Read-only.

Compiler may never modify Project.

---

# Validation Rules

A Project is considered valid only if:

- All required sections exist.
- No duplicated creative information exists.
- No compiler-specific information exists.
- Cross references are consistent.
- Schema version is supported.

---

# Alternatives Considered

## Prompt as Source

Rejected.

Reason:

Prompt cannot support iterative production.

Prompt duplicates information.

Prompt is model-dependent.

---

## Story Bible as Source

Rejected.

Reason:

Story is only one aspect of filmmaking.

Project contains story plus production data.

---

## Shot List as Source

Rejected.

Reason:

Shot List cannot represent global creative decisions.

# Compliance

Every future ADR must comply with ADR-001 unless explicitly superseded.

Every new subsystem must read Project.

No subsystem may establish another canonical data source.

---

# Design Principle

Project describes the film.

Compiler describes the model.

Prompt describes the execution.

Project is forever.

Prompt is disposable.
