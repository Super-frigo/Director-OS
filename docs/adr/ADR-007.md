# ADR-007: Retrieval Strategy for Library System

Date: 2026-07-11

## Status

Accepted

## Context

Director OS 的 Library 系统（ADR-004）存储结构化的电影知识条目，供 Engine 层在 Shot 设计阶段检索和参考。Composition Library 作为第一个实现的 Library，需要决定：**Engine 如何从 Library 中检索与当前 Shot 情绪上下文最相关的构图规则？**

两种候选策略：

**A. Keyword-based retrieval（关键词检索）**
- 维护一个受控词表（`_EMOTION_MAP`），将中/英文情绪词映射到标准化关键词
- 用 jieba 对中文查询分词，逐 token 查表取并集
- 与条目的 `emotions`/`scene_types`/`keywords` 字段做精确 token 匹配
- 100% 确定性，零外部依赖

**B. Embedding-based semantic retrieval（向量语义检索）**
- 用预训练模型（如 BAAI/bge-small-zh-v1.5）将查询和条目文本编码为向量
- 通过余弦相似度排序
- 可以处理词汇表外的表达（如隐喻、复合修饰）
- 需要 ~50MB 模型依赖，排序不透明

为做出决策，在 2026-07 进行了两轮对照实验：

1. **单 query 实验**（`隐隐不安`）：keyword 在修复 jieba 分词+简体中文词表后，semantic_score 从 0 恢复到正常水平。Embedding 跨语言（CN→EN）top-1 为 CIRCULAR（不相关），同语言（CN→CN）为 FRAME_WITHIN_FRAME（勉强相关）。

2. **9 query 扩展实验**：覆盖隐喻表达（细密的裂痕）、复合修饰（挥之不去的异样感、冰冷的仪式感）、外部类型片短语（逐渐逼近的恐惧、表面平静下的暗流）。结果：

| 方法 | top-1 相关命中 |
|------|--------------|
| Keyword (jieba + 受控词表) | **6/9 (67%)** |
| Embedding 跨语言 (CN→EN) | 2/9 (22%) |
| Embedding 同语言 (CN→CN) | 3/9 (33%) |

关键发现：
- Keyword 在负面情绪主导的 noir/thriller 场景中泛化良好，但存在 "ASYMMETRICAL 单一种植"问题（8/9 查询的 #1 都是它），这是条目内容 emotion 词汇不均衡导致的，非检索算法问题
- Embedding 同语言比跨语言仅提升 1/9，说明"跨语言稀释"不是主因——真正瓶颈是条目文本太短（~100词）且描述摄影技法而非情绪，导致 embedding 区分度极差（cosine 分数区间仅 0.04）
- Embedding 有一项 keyword 不具备的互补能力：正确识别了"无声的压迫→SYMMETRY"和"表面平静下的暗流→BALANCED"这两个隐喻映射

## Decision

**当前阶段采用 jieba 分词 + 受控词表（`_EMOTION_MAP`）的 keyword 检索作为 Library 系统的唯一检索策略。**

不引入 embedding-based retrieval。

### 具体实现

1. `_resolve_emotion_keywords()` 先做精确匹配，未命中则 jieba 分词后逐 token 查 `_EMOTION_MAP` 取并集
2. `_EMOTION_MAP` 同时维护简体中文和繁体中文条目，以及修饰词条目（隐隐、淡淡、微微等）
3. 评分函数为 `W_SEMANTIC * semantic_score + W_SHOT * shot_param_score`，两个权重默认为 0.5，可通过 `set_weights()` 调整

### 这不是"embedding 无用"的结论

本次实验结论是**在当前规模和实现条件下，keyword 的性价比更高**——不是 embedding 作为一种技术方向被否定。Embedding 在以下条件下可能变得更有竞争力：
- 条目文本长度增加到 200+ 词
- 条目文本从"摄影技法描述"改为"情绪优先描述"
- 使用专门的 cinematography-domain embedding 模型而非通用多语言模型
- 作为 recall 层（embedding 召回 top-10 → keyword 重排）而非独立排序器

## Re-evaluation Triggers

以下任一条件满足时，应重新评估是否引入 embedding 或混合检索：

1. **Library 条目规模达到 50+。** 更多条目 → 更大的 embedding 区分度（当前 14 条目下 cosine 分数挤在 0.04 区间内，50+ 条目应能拉开分数分布）

2. **Library 条目文本中文化且情绪化重写。** 如果条目的 `effect`/`notes` 字段从英文摄影技法描述改为中文情绪优先描述（每条 200+ 词），同语言 embedding 的排序质量预计显著提升。此条件下应重新运行 9-query 对照实验。

3. **出现 keyword 系统无法覆盖的新 modifier pattern。** 当前的 jieba 分词 + 受控词表覆盖了"修饰词+基础情绪词"模式（隐隐不安、淡淡忧伤等）。如果出现无法通过分词查表解决的新型中文情绪表达（如方言化、诗化、或纯隐喻无情绪词根的短语），且数量达到 3 个以上，应重新评估。

4. **出现非中文的情绪查询需求。** 当前 `_EMOTION_MAP` 覆盖中/英文，但如果未来 Project 中出现日语、韩语等情绪表达，且量足够大，应考虑用多语言 embedding 替代手工词表扩展。

## Alternatives Considered

### Alternative 1: Embedding-only retrieval

**Rejected for now.** 跨语言 embedding 仅 22% 准确率，同语言仅 33%，远低于 keyword 的 67%。引入 ~50MB 模型依赖 + 索引维护成本无法被当前的质量收益 justify。不排除未来重评。

### Alternative 2: Hybrid retrieval (embedding recall + keyword re-rank)

**Deferred.** 理论上是最优方案（embedding 的语义覆盖 + keyword 的精确性和透明性），但两个前提条件未满足：(a) embedding recall 的质量需要先提升（当前 top-10 recall 包含太多噪声），(b) 需要定义 re-rank 的融合策略。在 embedding 单独表现提升之前，hybrid 不会比纯 keyword 更好。

### Alternative 3: 不检索，Engine 自行推理

**Rejected.** 违反 ADR-004（Library Driven Knowledge System）。Engine 不应在每次 Shot 设计时重新发明构图知识。

## Consequences

### Positive

- 零外部依赖（仅需 jieba，已在项目中用于中文分词）
- 100% 确定性：相同输入 → 相同输出，可审计、可调试
- 透明性：每条匹配结果都可以追溯到具体的 token → map 映射链路
- 维护成本低：新增情绪覆盖只需在 `_EMOTION_MAP` 中加一行

### Negative

- 受控词表覆盖率是硬天花板：任何未收入 map 的新表达方式都是盲区（虽可通过 jieba 分词缓解，但不能根除）
- 存在"单一种植"风险：词汇覆盖最广的条目（当前是 ASYMMETRICAL）会在多数查询中排第一，需要定期审计和 rebalance 条目的 `emotions` 字段
- 无法处理纯隐喻表达（如"细密的裂痕"）——这类表达没有可分词的情绪词根，keyword 只能通过 shot-param 得分获得排名，semantic_score 仍为 0
- 扩展到非中/英文需要新增整张词表

### Neutral

- 评分权重（W_SEMANTIC / W_SHOT）是可调的，但默认值 0.5/0.5 基于直觉而非系统校准。未来可以通过 cinematographer 标注数据来学习最优权重

## Implementation Notes

- 检索逻辑位于 `libraries/composition/query.py`
- 受控词表位于 `query.py` 内的 `_EMOTION_MAP` 和 `_EMOTION_MODIFIER_MAP`
- 对照实验脚本和报告位于 `experiments/` 目录
- 测试覆盖位于 `tests/test_composition_query.py`（45 个用例，含 8 个回归测试）
